# Medical Records Sorter Assist v0.9 Design Spec

## Purpose

v0.9 turns the client proof-of-concept into a cleaner, reviewer-facing UI that sits on top of the v0.8.6 UI run artifact contract.

The UI should preserve the client's core workflow:

```text
Received Medical Records
vs
ERE Medical Records
```

The UI should not feel like a generic engineering dashboard. It should feel like a workbench for reviewing duplicate, near-duplicate, and overlapping medical record pages.

## Product promise

The system helps staff review candidate duplicate pages faster by showing likely matches side by side, explaining why they were flagged, and saving reviewer decisions.

## Non-promise

The system does not automatically remove records, delete pages, or make final legal/medical determinations.

## Primary reviewer workflow

1. Load a run generated by the engine.
2. See a Received vs ERE batch summary.
3. Review candidate matches.
4. Compare pages side by side.
5. Assign a reviewer decision.
6. Save decisions to `review_decisions.json`.
7. Export CSV or decisions JSON.

## What to keep from the original POC

- Two-bucket mental model.
- Received records compared against ERE records.
- Context overlaps vs image duplicates.
- Side-by-side page comparison.
- CSV/export behavior.
- Local/private positioning.

## What v0.9 improves

- Cleaner typography and spacing.
- Denser candidate queue.
- Clear reviewed/unreviewed state.
- Human decision buttons.
- Persistent `review_decisions.json` writes.
- OCR / Tesseract / vision fallback / embeddings / LLM capability status.
- Internal diagnostics separated from normal review flow.

## Primary screens

### Batch summary

Show the familiar comparison frame:

- Received Medical Records
- ERE Medical Records

For v0.9 this is derived from the loaded run folder rather than live uploads.

### Review queue

Primary filters:

- Needs Review
- Likely Duplicates
- Partial Overlaps
- Reviewed
- All

Secondary match filters:

- All match types
- Context overlaps
- Image duplicates
- OCR-based
- Vision fallback expected

### Candidate review

Show:

- Received page preview
- ERE page preview
- File names
- Page numbers
- Engine label
- Confidence
- Match type
- OCR/vision status
- Explanation
- Signals
- Reviewer decision buttons
- Reviewer note

### Diagnostics

Keep diagnostics available but hidden by default. Include:

- engine version
- artifact contract version
- truth status
- layer recall, when present
- OCR summary
- AI call summary

## Decision labels

Allowed human labels:

```text
duplicate
likely_duplicate
possible_duplicate
partial_overlap
not_duplicate
needs_review
```

Decision shape:

```json
{
  "candidate_id": "cand_000123",
  "human_label": "duplicate",
  "reviewer_note": "Same page, ERE version is lower scan quality.",
  "reviewed_at": "2026-05-28T00:00:00Z"
}
```

## Visual direction

- Clean, calm, professional.
- Use color for status and source identity, not decoration.
- Received records: blue.
- ERE records: indigo.
- Duplicate/complete: green.
- Needs review/possible: amber.
- Image/high-risk issue: rose.
- Dense enough for actual review work.

## Accessibility baseline

- Keyboard-focusable controls.
- Visible focus states.
- Text labels on buttons.
- Color is not the only status indicator.
- Body text should meet readable contrast.

## v0.9 done criteria

v0.9 is done when:

1. `dupe-engine review-ui --run-dir <run-dir>` opens the local UI.
2. The UI loads v0.8.6 artifacts.
3. The UI shows Received vs ERE framing.
4. The UI shows capability status.
5. The UI lists candidates with filters.
6. The UI opens side-by-side page previews.
7. The UI displays why the candidate was flagged.
8. The UI saves reviewer decisions to `review_decisions.json`.
9. Saved decisions persist after reload.
10. The UI exports CSV and decisions JSON.

## Out of scope for v0.9

- Multi-user auth.
- Role permissions.
- Cloud deployment.
- Database-backed storage.
- Full audit history.
- Live embeddings.
- Live LLM adjudication.
- Automated record deletion.
- Case management integration.
