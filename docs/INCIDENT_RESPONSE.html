Application
Container crashes and restarts
Self-heal
Automated --restart unless-stopped brings it back in <30s. CloudWatch alarm fires if it stays down.
If it restarts more than 3× in 5 minutes
  • SSH into instance: ssh ec2-user@<host>
  • Check logs: docker logs dupe-engine-review --tail 100
  • If OOM (out of memory): upgrade to t3.large
  • If Python exception: call Dustin — patch needed
Handles: Automated → if repeated: Call Dustin

Application
OpenAI / Bedrock API throttled
Self-heal
Automated Engine retries on rate limit errors. Jobs slow down but complete. No intervention needed unless a job is stuck for >30 minutes.
If a job is stuck >30 min
  • In the review UI: cancel the stuck job and resubmit
  • Or restart container: docker restart dupe-engine-review
  • Check API quota in OpenAI dashboard if repeated

Infrastructure
EBS disk approaching full
Runbook
Detection CloudWatch alarm fires at 80% disk usage. Anyone with SSH can clear space using the steps below.
Fix steps
  • SSH in: ssh ec2-user@<host>
  • Check usage: df -h /data
  • Delete old page images (safe — regenerated on demand): find /data/runs -name "page_images" -type d -exec rm -rf {} + 2>/dev/null || true
  • Verify space freed: df -h /data
  • If still full, extend EBS volume in AWS Console → EC2 → Volumes → Modify

Application
Reviewers locked out (auth token)
Runbook
Symptom Review UI shows "Invalid token" or token overlay won't accept the password. No data at risk.
Fix steps
  • SSH in: ssh ec2-user@<host>
  • Edit the env file: sudo nano /srv/apps/dupe-engine/.env
  • Set or correct: DUPE_UI_AUTH_TOKEN=<correct-token>
  • Restart container: docker restart dupe-engine-review
  • Confirm with reviewers: the token overlay will accept the new token

Infrastructure
CPU throttling / jobs very slow
Runbook
Cause t3 instances are burstable. CPU credit exhaustion throttles the instance to baseline (20% of a vCPU). Jobs still complete — just slowly.
Fix steps
  • Check CPU credits in CloudWatch: EC2 → Metrics → CPUCreditBalance
  • If credits near zero: upgrade instance type in AWS Console → EC2 → Stop instance → Change instance type → t3.large → Start
  • EBS data survives a stop/start — no data loss
Handles: Runbook

Network
Security group blocks port 8765
Runbook
Symptom Reviewers on VPN can't reach the UI. Server is running fine — only external access is blocked.
Confirm it's the security group
  • SSH into instance and confirm server is up: curl http://localhost:8765/api/status
  • If that returns {"ok": true} — it's a network/SG issue, not the app
Fix in AWS Console
  • EC2 → Instances → select instance → Security tab → click the security group
  • Inbound rules → Edit → Add rule: TCP 8765 from VPN CIDR
  • Save. Immediate effect — no restart needed.
Handles: Runbook

Infrastructure
EC2 instance unreachable
Call Dustin
Detection CloudWatch StatusCheckFailed alarm → SNS → text to Dustin's phone. Instance is down or unresponsive at the hypervisor level.
What to do while waiting for Dustin
  • In AWS Console: EC2 → Instances → check status. If "stopped": Start it.
  • If "running" but StatusCheck failing: reboot from Console → Actions → Reboot
  • If still failing after reboot: wait for Dustin — may need new instance
After instance is back
  • Verify EBS still attached: check Volumes in Console
  • Start container if it didn't auto-restart: docker start dupe-engine-review
  • Confirm: curl http://localhost:8765/api/status

Application
Corrupt job state / broken results
Call Dustin
Symptom A job's results page shows errors or the UI crashes loading a specific run. The healing harness can repair some cases.
Attempt self-repair first
  • SSH in and run: dupe-engine heal --run-dir /data/runs/<job-id> --verbose
  • If it reports "HEALED" or "IMPROVED" — the run is usable again
  • If it reports "RESISTANT" — call Dustin with the job ID and the error from docker logs

Network
Reviewer can't connect (VPN side)
Runbook
Key test If the server is reachable via SSH, the server is fine. VPN issues are on the client or IT side.
Triage steps
  • Confirm server is up: curl http://localhost:8765/api/status from SSH → should return {"ok": true}
  • Ask the reviewer: are they on VPN? Can they reach any other internal resource?
  • If VPN is the issue: direct them to IT helpdesk — this is outside our stack
  • If multiple reviewers are affected: check the security group first (see card above)

Infrastructure — Critical
AWS Availability Zone outage
Critical
Impact Full instance down. Single-AZ deployment means no automatic failover. Data is on EBS — tied to the same AZ. S3 backups are cross-region and safe.
Recovery (Dustin + AWS)
  • Check AWS Service Health Dashboard for the AZ outage — gives ETA
  • If outage is extended (>2h): create a snapshot of the EBS volume from a healthy AZ
  • Launch a new EC2 instance in a different AZ, restore from snapshot
  • Restore latest S3 backup to /data if EBS is not recoverable
  • Restart container with same configuration
Handles: Call Dustin AWS Support

Data — Critical
Run artifacts accidentally deleted
Critical
Recovery path S3 backups retained 90 days. Page images are regeneratable — the irreplaceable data is results.json, false_negatives.csv, and review_decisions.json.
Restore from S3 backup
  • List available backups: aws s3 ls s3://your-backup-bucket/dupe-engine/
  • Download closest backup: aws s3 cp s3://your-backup-bucket/dupe-engine/runs_backup_YYYYMMDD.tar.gz /tmp/
  • Extract to /data/runs: tar xzf /tmp/runs_backup_YYYYMMDD.tar.gz -C /
  • Fix permissions: sudo chown -R ec2-user:ec2-user /data
  • Restart container
